## 一、代码质量

### 🔴 高优先级

#### 1.1 TypeScript 未启用 strict 模式

**文件**: `tsconfig.json`  
**当前状态**: 未启用 `strict`、`noUnusedLocals`、`noUnusedParameters`、`noImplicitReturns` 等

```json
// 当前配置 - 缺少这些关键选项:
{
  "compilerOptions": {
    "target": "ES2022",
    "skipLibCheck": true,
    // ❌ 缺少 strict: true
    // ❌ 缺少 noUnusedLocals: true
    // ❌ 缺少 noUnusedParameters: true
    // ❌ 缺少 noImplicitReturns: true
    // ❌ 缺少 noFallthroughCasesInSwitch: true
  }
}
```

**影响**: `strictNullChecks` 关闭 → 潜在 null/undefined 引用风险；`noImplicitAny` 关闭 → 允许隐式 any。虽然 `tsc --noEmit` 通过，但大量潜在问题被掩盖。  
**修复建议**: 逐步启用 strict 模式。先从 `strictNullChecks` + `noImplicitAny` 开始，修复暴露的问题后再启用 `noUnusedLocals`。

## 二、性能

### 🔴 高优先级

#### 2.1 无代码分割 — 所有页面组件静态加载

**文件**: `src/App.tsx`（第 6-25 行）

12 个页面组件全部使用静态 `import`，`React.lazy` / `Suspense` 搜索结果为 **0**：

```tsx
// 当前 - 静态导入全部12个组件
import AdminPanel from './components/AdminPanel';    // 78.6KB
import BracketPage from './components/BracketPage';
import HistoryHallPage from './components/HistoryHallPage';
import HomeTab from './components/HomeTab';
import LeaderboardTab from './components/LeaderboardTab';
import MatchDetailPage from './components/MatchDetailPage';  // 58.1KB
import MatchesTab from './components/MatchesTab';
import MeTab from './components/MeTab';

```

**影响**: 即使访问首页，也会下载所有页面（包括 78KB 的 AdminPanel、58KB 的 MatchDetailPage 等），首次加载体积巨大。  
**修复建议**:
```tsx
const AdminPanel = React.lazy(() => import('./components/AdminPanel'));
const BracketPage = React.lazy(() => import('./components/BracketPage'));
const LeaderboardTab = React.lazy(() => import('./components/LeaderboardTab'));
// ... 对非首屏页面使用 lazy
```

#### 2.2 Vite 构建缺少 manualChunks 配置

**文件**: `vite.config.ts`（第 6-33 行）

当前无任何 `build.rollupOptions` 配置，第三方依赖全打一个 chunk：

```ts
export default defineConfig(() => {
  return {
    plugins: [react(), tailwindcss()],
    // ❌ 缺少 build.rollupOptions.output.manualChunks
    server: { /* ... */ },
  };
});
```

**修复建议**: 将大型依赖分离为独立 chunk：
```ts
build: {
  rollupOptions: {
    output: {
      manualChunks: {
        'vendor-react': ['react', 'react-dom'],
        'vendor-animation': ['motion', 'gsap'],
        'vendor-charts': ['recharts'],
        'vendor-socket': ['socket.io-client'],
        'vendor-canvas': ['html2canvas'],
      },
    },
  },
},
```

#### 2.3 排行榜接口性能瓶颈 — O(n × (P+T))

**文件**: `src/server/routes/matches.ts`（第 426-548 行，`GET /api/leaderboards`）

对每个用户执行遍历全部 predictions 和 transactions：

```ts
// 每个用户执行的操作：
const latestTransaction = db.transactions
  .filter((item) => item.userId === user.id)  // 第443行 - 每用户扫全量交易
  .sort().find();
const userPredictions = db.predictions.filter(p => p.userId === user.id); // 第457行
const totalCount = userPredictions.filter(p => p.status === 'SETTLED').length;  // 第458行
const wonCount = userPredictions.filter(p => p.result === 'WON').length;  // 第459行
// ... 还有 4 次类似的 filter/reduce 操作
```

**复杂度**: N 个用户 × (P 条预测 + T 条交易)，群规模增长时性能急剧下降。  
**修复建议**: 使用 Map/索引一次遍历完成所有用户统计：
```ts
const predictionMap = new Map<string, Prediction[]>();
for (const p of db.predictions) {
  if (!predictionMap.has(p.userId)) predictionMap.set(p.userId, []);
  predictionMap.get(p.userId)!.push(p);
}
// 然后对每个用户直接从 Map 取值，避免重复 filter
```

## 三、架构设计

### 🔴 高优先级

#### 3.1 `room-1` 硬编码分布在 8+ 处

| 文件 | 行号 | 严重度 | 说明 |
|------|------|--------|------|
| `src/db/db_service.ts` | 336 | **高** | `return room?.id \|\| 'room-1'` — 单房间 fallback |
| `src/server/ai.ts` | 83 | **高** | AI 服务房间 fallback 默认值 |
| `src/db/db_service.ts` | 186/205/224 | 中 | 种子预测数据 `groupId: 'room-1'` |

**影响**: 如果 `db.rooms` 为空数组，系统静默 fallback 到 `'room-1'`（可能不存在的 ID），后续查询可能失败。多房间功能被硬编码限制。  
**修复建议**: 
1. 代码中 fallback 值改为从配置读取或抛出明确错误
2. `initial_data.ts` 和种子预测数据中加上注释 `// TODO: 多房间部署时需更新`

#### 3.2 `db_service` 被 24 个文件导入（上帝对象模式）

`dbService` 单例直接被 24 个文件导入，任何模块都可以随意读写数据。  
**修复建议**: 
- 考虑为高频读取路径提供只读 snapshot
- 将写操作收口到 service 层，route 层只读不写
- 长期考虑 repository 模式

## 四、功能稳定性

### 🔴 高优先级

#### 4.1 结算服务缺少事务性保证

**文件**: `src/server/services/settlement_service.ts`

结算流程涉及多步数据变更（更新 prediction 状态、写入 transaction、更新 wallet、触发 badge/activity），在 JSON 文件存储模式下，这些操作虽然有写锁，但如果中间某步失败，前面的变更已写入 db.json，无法回滚。

**修复建议**: 
1. 结算前自动备份（已有部分实现）
2. 在结算前验证数据完整性
3. 迁移 MySQL 后使用数据库事务

---

### 🟡 中优先级

#### 4.2 catch 块仅 console.error 未向上传播

部分异步操作中的 catch 块仅打印错误而不向调用方返回错误状态，导致前端无法感知失败。

#### 4.3 AI 服务不稳定时的降级处理

当所有 AI 提供商均不可用时（`ai.ts` 多提供商路由），前端显示"AI 内容暂不可用"已实现。但部分场景下（如首页 AI 推荐）会显示空白区域而非友好的降级提示。
