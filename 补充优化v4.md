# SJB World Cup Betting｜世界杯档案馆与球队资料增强实施计划

> 适用仓库：`https://github.com/klpk-cgt/SJB-WorldCup-Betting.git`  
> 基础分支：以 Codex 执行时仓库默认分支的最新提交为准  
> 项目定位：2026 世界杯朋友群观赛、虚拟积分竞猜、排行榜与 AI 娱乐互动平台  
> 本任务目标：在不破坏现有赛程、竞猜、排行榜和后台功能的前提下，增加完整的球队资料、球员阵容、世界杯历史长廊和每日历史问答能力。

---

## 1. 已确认的产品决策

1. 球队详情及历史资料允许游客访问。
2. 球员身价使用静态数据，不做实时同步。
3. 第一版覆盖全部 48 支参赛球队，不只维护热门球队。
4. 球员详情第一版使用移动端底部抽屉，不新建独立球员详情路由。
5. 历史长廊覆盖历届世界杯基础资料，重点届次补充更完整故事。
6. 第一版暂不增加“支持球队 / 加入球队阵营”功能。
7. FIFA 排名使用静态快照，不做自动更新，重点保证加载稳定。
8. 历史资料需要接入现有每日答题题库。
9. 所有新功能仍保持“虚拟娱乐积分，不涉及真实资金”的产品定位。
10. 不在底部导航增加第六个入口，保持现有：首页、赛程、竞猜、排行榜、我的。

---

## 2. 项目现状与实施原则

### 2.1 当前已有能力

- React 19 + Vite + Tailwind CSS v4 移动端前台
- Express 全栈服务
- 首页焦点战
- 104 场世界杯赛程
- 比赛详情、比赛事件、首发阵容、比赛统计
- 群友倾向、AI 赛前 / 赛后分析
- 虚拟积分竞猜、排行榜、管理后台
- 每日问答
- API-Football、The Odds API 与 AI 提供商同步框架
- JSON 文件持久化

### 2.2 实施原则

1. **资料嵌入现有流程**：从首页、赛程和比赛详情自然进入，而不是另建复杂资讯门户。
2. **静态优先，稳定优先**：排名、参赛次数、历届最佳、身价、球员资料使用仓库内静态数据。
3. **当前赛事与历史档案分离**：现有比赛数据负责本届赛程；独立静态模块负责历史和百科。
4. **事实与 AI 分离**：排名、身价、冠军次数等不能由 AI 生成；AI 只负责总结和趣味表达。
5. **移动端优先**：球队详情使用标签页，球员详情使用底部抽屉。
6. **渐进实施**：先完成球队基础档案，再做球员、历史长廊和题库整合。

# 6. 静态数据目录

新增：

```text

├── historicalMatches.ts

```

推荐使用 TypeScript 数据文件，便于类型检查。

## 6.1 数据加载要求

- 静态数据随构建产物加载。
- 不依赖运行时第三方 API。
- 不允许每次打开球队页请求第三方网站。
- 数据缺失时显示“资料待补充”，不能白屏。
- 数值字段统一使用格式化函数。
- 每份资料保存快照日期和来源说明。

## 6.2 身价数据要求

```ts
marketValueEur: 180_000_000,
marketValueSnapshotDate: '2026-05-31',
dataSource: '静态资料快照'
```

页面显示：

```text
参考身价：€180m
数据快照：2026-05-31
```

禁止称为“实时身价”“官方身价”。

## 6.3 参考项目复用规则

参考项目：

```text
https://github.com/oyvhov/world-cup-pool
```

可以参考：

- 数据结构
- 页面布局
- 球队/球员字段设计
- 锦标赛预测交互
- 数据导入方式

执行前必须：

1. 检查参考仓库是否真的包含相关数据。
2. 检查数据是否来自受限制的第三方来源。
3. 注意参考仓库为 GPL-3.0，直接复制代码可能影响当前项目许可证。
4. 不直接整段复制 GPL 代码，应重新实现。
5. 身价数据尤其需要记录原始来源和快照日期。
6. 在 `sources.ts` 记录每组数据来源、日期和备注。

推荐策略：参考项目只作为结构和交互参考，事实数据自行整理；身价做一次性静态录入。

## 产品结论

本次增强不是把项目改造成普通足球百科，而是建立一层贯穿赛程、比赛详情、竞猜和每日问答的“世界杯档案馆”。

核心路径：

```text
看到比赛
→ 点击球队
→ 了解实力、阵容和历史
→ 返回比赛
→ 参与娱乐竞猜
→ 通过每日问答继续探索世界杯
```
