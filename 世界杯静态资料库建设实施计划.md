# SJB WorldCup Betting 世界杯静态资料库建设实施计划

## 一、建设目标

当前项目已经具备球员头像、球员信息、赛事竞猜、排行榜、AI 分析等基础能力。下一阶段需要补强世界杯信息内容，尤其是球队详情页中的历史资料，包括：

1. 球队基础资料
2. 历届世界杯战绩
3. 2026 世界杯预选赛战绩
4. 历史交锋
5. 球队故事卡片
6. 世界杯历史内容卡片
7. 数据来源标注和人工校验状态

本模块的目标是建立一套可维护、可扩展、适合朋友群娱乐展示的静态资料库，让项目不仅能竞猜，也能作为一个轻量级世界杯信息站使用。

------

## 二、设计原则

### 1. 静态数据优先

本模块不依赖实时 API。

原因：

1. 球队历史战绩、历届世界杯表现、历史交锋等属于历史信息，不需要频繁更新。
2. 静态数据更稳定，适合长期维护。
3. 可以减少 API 成本和接口不稳定风险。
4. 方便后续部署到云服务器。
5. 方便人工校验和来源标注。

### 2. 数据准确性优先

所有涉及具体数字的数据，例如：

1. 世界杯参赛次数
2. 冠军次数
3. 历届世界杯成绩
4. 预选赛胜平负
5. 历史交锋次数
6. 进球数
7. 最佳成绩

必须人工核对，不允许完全由 AI 直接生成。

AI 只能用于：

1. 把已确认的数据整理成简介文案。
2. 生成一句话总结。
3. 生成球队标签。
4. 生成历史故事卡片文案。
5. 统一中文表达风格。

AI 不能作为事实来源。

### 3. 数据来源需要分级

资料来源建议分为 4 级：

#### A 级：官方来源

优先用于核对关键数据。

包括：

1. FIFA 官网
2. UEFA 官网
3. AFC 官网
4. CAF 官网
5. CONMEBOL 官网
6. CONCACAF 官网
7. OFC 官网

适合核对：

1. 2026 预选赛结果
2. 晋级方式
3. 官方比赛结果
4. 小组排名
5. 赛程时间
6. 官方统计数据

#### B 级：历史资料来源

适合整理世界杯历史资料。

包括：

1. RSSSF
2. 世界杯历史资料库
3. 各类长期维护的足球历史统计站

适合整理：

1. 历届世界杯战绩
2. 世界杯历史排名
3. 历届比分
4. 世界杯预选赛历史资料
5. 球队历届参赛记录

#### C 级：百科类来源

适合快速补充结构化资料。

包括：

1. Wikipedia
2. Wikidata

适合整理：

1. 球队世界杯历史
2. 球队基础资料
3. 球队历史简介
4. 球员基础资料
5. 历届世界杯概览
6. 经典比赛背景

#### D 级：新闻和体育媒体来源

适合补充故事性内容。

包括：

1. ESPN
2. Reuters
3. The Athletic
4. The Guardian
5. 其他体育媒体

适合整理：

1. 球队故事
2. 传统恩怨
3. 焦点对决背景
4. 名场面介绍
5. 球员故事

## 三、数据准确性等级

每条资料需要增加准确性等级，方便后续维护。

```ts
type AccuracyLevel = 'verified' | 'needs_review' | 'summary_only';
```

含义：

```text
verified：已人工核对，可以展示具体数字。
needs_review：来源待复核，可以展示但后台标记。
summary_only：只展示概述，不展示精确数字。
```

使用规则：

1. 精确数字必须尽量使用 `verified`。
2. 存在口径争议的数据，例如全部历史交锋总数，建议使用 `needs_review`。
3. 无法确认数字时，只展示概述，使用 `summary_only`。
4. 前端不需要强制展示准确性等级，但后台或数据维护页可以保留字段。


推荐使用 TypeScript 文件，而不是纯 JSON，原因：

5. 能减少拼写错误和字段缺失。

## 十、AI 使用规范

### 10.1 AI 可以做的事

AI 可以用于：

1. 生成球队简介。
2. 生成球队标签。
3. 把长资料压缩成短卡片。
4. 统一中文表达风格。
5. 生成“AI 一句话看点”。
6. 根据已确认数据生成概括文案。
7. 生成页面展示文案。

### 10.2 AI 不允许做的事

AI 不允许直接编写以下事实数据：

1. 历史交锋次数。
2. 历届世界杯成绩。
3. 预选赛战绩。
4. 比赛比分。
5. 球员身价。
6. 球队冠军次数。
7. 世界杯参赛次数。
8. 任何具体数字。

### 10.3 推荐 AI 工作流

```text
人工整理事实数据
  ↓
人工核对关键数字
  ↓
AI 生成 summary / tags / story card
  ↓
人工快速检查文案
  ↓
写入静态数据文件
```

## 十四、数据维护规范

### 14.1 不要把大段复制内容直接放进项目

历史资料可以参考公开来源，但项目里应使用整理后的短内容。

每条内容建议控制：

1. 简介：100-200 字以内。
2. 卡片内容：50-150 字以内。
3. AI 简评：50 字以内。
4. 故事卡片：150 字以内。

### 14.2 所有关键数字都要有来源

以下字段必须尽量标注来源：

1. 世界杯参赛次数
2. 世界杯冠军次数
3. 历届世界杯成绩
4. 预选赛战绩
5. 历史交锋
6. 纪录类数据
7. 金靴榜
8. 经典比赛比分

### 14.3 历史交锋要避免过度承诺

如果不同来源口径不统一，不要写绝对结论。

推荐写法：

```text
公开资料统计中，两队历史交锋次数因统计口径不同可能存在差异。本页面优先展示世界杯交锋和近期正式比赛。
```

不推荐写法：

```text
两队历史交锋一定是 113 场。
```

## 十五、风险点

### 15.1 数据错误风险

风险：

AI 或人工整理时可能写错历史战绩。

解决：

1. 数据字段增加来源。
2. 增加准确性等级。
3. 先做样例球队。
4. 关键数字人工复核。
5. 需要复核的数据标记 `needs_review`。

------

### 15.2 历史交锋口径争议

风险：

不同网站统计口径不同。

解决：

1. 优先展示世界杯交锋。
2. 全部交锋只做概览。
3. 页面增加口径说明。
4. 不确定时不展示具体总数。

------

### 15.3 数据文件过大

风险：

48 队数据和历史卡片过多，导致文件变大。

解决：

1. 按球队拆分文件。
2. 使用 `index.ts` 统一导出。
3. 页面按需读取。
4. 后续可考虑懒加载。

------

### 15.4 页面信息过载

风险：

球队详情页内容太多，用户看不完。

解决：

1. 使用 Tab。
2. 重点信息放顶部。
3. 卡片化展示。
4. 每段内容短。
5. 长资料折叠。

------
