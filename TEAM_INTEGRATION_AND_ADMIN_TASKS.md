# 球队资料接入与后台管理升级实施文案

更新时间：2026-06-07

## 1. 目标说明

当前项目已经具备：

- 赛程展示
- 赔率同步
- 竞猜下注
- AI 内容生成
- 分享卡生成
- 基础后台管理

下一阶段建议把项目升级为“既能竞猜，也能看内容、聊内容、运营内容”的世界杯群聊产品。

本阶段重点分为两条线：

1. 接入球队资料内容能力  
包括球队概况、球员名单、历史战绩。

2. 升级后台管理能力  
让后台不只是“改比分、调积分”，还要能做内容管理、同步诊断、AI 运营和活动配置。

---

## 2. 总体实施原则

- 不直接照搬外部项目整套前后端，只吸收有价值的数据结构和内容模块
- 保留当前项目的竞猜主链路，球队资料作为增强内容层接入
- 第一阶段优先做“可展示、可管理、可联动”
- 第二阶段再做“更精细的数据质量、AI 运营和自动化能力”
- 所有新内容都优先适配移动端和群聊传播场景

## 3.3 数据导入与清洗任务

### 任务 B1：建立外部数据映射脚本

目标：把 `chuifei/world-cup-2026` 的球队内容转成当前项目可用格式。

建议做法：

- 新建一个数据转换脚本
- 把外部仓库的 `teams / players / history` 转成当前项目自己的 JSON 或 seed
- 建立球队 code 映射表，避免国家名、中文名、缩写不一致

建议输出：

- `team_profiles.json`
- `team_players.json`
- `team_history.json`

验收标准：

- 数据可重复导入
- 同一支球队不会产生重复记录
- 名称映射错误可在日志中看到

### 任务 B2：做内容质量分级

目标：区分“可信基础字段”和“估算扩展字段”。

建议分层：

- `core`：国家名、主教练、阵型、世界杯成绩
- `extended`：身价、惯用脚、能力值、简介

验收标准：

- 前台优先使用 `core` 数据
- `extended` 字段缺失时不影响页面主流程

## 6. 验收口径

完成本阶段后，项目应达到以下效果：

- 用户在看比赛和下注时，可以随手查看球队概况、阵容和历史战绩
- AI 文案不再只像赔率解读，而更像真实赛事内容
- 分享卡更有讨论点，适合发到群里
- 后台不再只是手工改比分工具，而是具备内容管理、AI 运营、同步诊断的完整能力

---

## 7. 我建议的实际落地顺序

如果按你当前项目状态继续推进，我建议这样做：

1. 先补数据结构和 seed
2. 再做球队详情页
3. 再做后台球队资料管理
4. 再做 AI 内容联动
5. 最后做导入工具和高级运营能力

这个顺序的好处是：

- 前台很快能看到成果
- 后台不会一上来就做太重
- 能先验证用户是否真的会点击和讨论球队内容
