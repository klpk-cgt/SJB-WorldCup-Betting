# SJB WorldCup Betting 朋友群世界杯娱乐社区优化实施计划

## 一、项目优化目标

当前项目已经具备世界杯赛事竞猜、排行榜、AI 分析、赔率展示、管理后台等基础能力。下一阶段目标不是单纯增加功能，而是将项目优化成一个：

> 朋友群真的愿意天天打开、比赛日愿意互动、管理员维护不累的小型世界杯娱乐社区。

项目核心用户不是陌生公网用户，而是朋友群成员。因此整体优化方向应围绕以下目标展开：

1. 让用户觉得这是“我们群自己的世界杯网页”。
2. 让比赛日前、中、后都有打开网页的理由。
3. 让竞猜不仅是积分输赢，也能产生群内调侃和互动。
4. 让 AI 输出短、有趣、适合群内传播。
5. 让管理员可以轻松维护用户、比赛、积分、竞猜、结算和数据备份。
6. 暂不引入真实金钱、充值、提现、实物奖品等复杂或敏感功能。
7. 当前阶段继续兼容现有 `db.json` 数据模式，后续正式部署前再考虑 MySQL 迁移。

## 三、不做范围

本轮暂不实现以下内容，避免项目复杂度过高：

1. 不做真实金钱充值、提现、兑换。
2. 不做微信 OAuth 登录。
3. 不做公开注册系统。
4. 不做复杂聊天室。
5. 不做直播视频。
6. 不做多群组/多房间完整隔离系统。
7. 不在本轮强制迁移 MySQL。
8. 不接入复杂爬虫。
9. 不做 APP 打包。
10. 不把竞猜道具设计成可签到领取或可刷取奖励的经济系统。

## 四、核心设计原则

### 1. 娱乐优先，但保持排行榜公平

项目是朋友群娱乐竞猜，不涉及真实金钱。所有文案应使用：

- 娱乐竞猜
- 娱乐倾向
- 积分玩法
- 群内娱乐排行
- AI 娱乐分析

避免使用：

- 投资建议
- 稳赚
- 推荐下注
- 真实赔率收益
- 现金收益

### 2. 竞猜卡系统要公平

竞猜卡不是签到奖励，不是任务奖励，也不是管理员随意补发的刷榜工具。

本轮规则：

1. 每个登录账号默认获得每种竞猜卡各 1 张。
2. 竞猜卡作为账号初始标配。
3. 不通过签到领取。
4. 不通过每日任务领取。
5. 不通过连续登录领取。
6. 不支持购买。
7. 不支持用户之间转赠。
8. 每场比赛最多只能使用 1 张竞猜卡。
9. 每一张卡使用后即消耗。
10. 使用竞猜卡的比赛，需要在竞猜记录和积分流水中清晰标记。
11. 排行榜统计中需要保留“是否使用竞猜卡”的记录，保证可追溯。

### 3. 所有重要操作必须可追溯

以下行为必须产生记录：

- 用户下注
- 用户修改竞猜
- 用户使用竞猜卡
- 比赛结算
- 积分变化
- 管理员调账
- 管理员修改比分
- 管理员锁定/解锁比赛
- AI 内容生成
- 数据备份导出

## 七、功能模块三：竞猜道具卡系统

### 7.1 功能目标

增加竞猜趣味性，但必须兼顾公平性和排行榜可信度。

本轮只实现 4 种竞猜卡：

1. 免亏卡
2. 双倍卡
3. 反悔卡
4. 保底卡

### 7.2 核心公平规则

必须严格实现：

1. 每个用户账号默认获得每种竞猜卡各 1 张。
2. 用户创建时自动发放。
3. 老用户首次进入或首次迁移时自动补齐。
4. 不通过签到领取。
5. 不通过任务领取。
6. 不通过连续登录领取。
7. 不开放购买。
8. 不支持转赠。
9. 一场比赛最多只能使用 1 张竞猜卡。
10. 一张竞猜卡只能使用一次。
11. 使用后必须记录到竞猜记录、积分流水和群内动态。
12. 管理后台可查看用户竞猜卡使用情况。
13. 管理员不建议随意补发竞猜卡；如保留补发功能，必须写入管理员操作日志。

### 7.3 卡片类型与规则

#### 免亏卡

效果：

```text
本场输了返还 50%
```

结算规则：

- 如果该竞猜最终亏损，则返还亏损金额的 50%。
- 如果该竞猜盈利或走平，不触发返还。
- 返还记录必须写入积分流水。
- 群内动态展示“使用免亏卡减少亏损”。

#### 双倍卡

效果：

```text
本场赢了收益 ×2
```

结算规则：

- 只对盈利部分翻倍。
- 本金不重复翻倍。
- 如果竞猜失败，不触发收益。
- 如果竞猜走平，不触发收益。
- 群内动态展示“使用双倍卡收益翻倍”。

#### 反悔卡

效果：

```text
开赛前 10 分钟可修改一次
```

使用规则：

- 只允许在开赛前 10 分钟之前使用。
- 已锁定比赛不可使用。
- 每张反悔卡只允许修改一次竞猜。
- 修改前后内容必须记录。
- 群内动态展示“使用反悔卡修改竞猜”。

建议：

- 反悔卡不直接影响结算收益，但会影响最终竞猜内容。
- 反悔卡使用后即消耗。

#### 保底卡

效果：

```text
本场最多亏 1000 分
```

结算规则：

- 如果本场实际亏损超过 1000 分，则最多只扣 1000 分。
- 如果实际亏损小于等于 1000 分，则不额外生效。
- 如果盈利，不触发。
- 群内动态展示“使用保底卡限制最大亏损”。

### 7.4 数据结构建议

```ts
type PredictionCardType = 'loss_refund' | 'double_profit' | 'edit_once' | 'loss_cap';

type UserPredictionCard = {
  id: string;
  userId: string;
  type: PredictionCardType;
  status: 'available' | 'used';
  grantedReason: 'account_default' | 'admin_adjustment';
  grantedAt: string;
  usedAt?: string;
  usedMatchId?: string;
  usedPredictionId?: string;
};

type PredictionCardUsage = {
  id: string;
  userId: string;
  matchId: string;
  predictionId: string;
  cardId: string;
  cardType: PredictionCardType;
  effectSummary: string;
  createdAt: string;
};
```

在 `db.json` 中新增：

```ts
userPredictionCards: UserPredictionCard[];
predictionCardUsages: PredictionCardUsage[];
```

### 7.5 用户创建时发放规则

新增服务：

```text
src/services/prediction_card_service.ts
```

需要实现：

```ts
grantDefaultCardsToUser(userId: string): void;
ensureDefaultCardsForExistingUser(userId: string): void;
getUserAvailableCards(userId: string): UserPredictionCard[];
usePredictionCard(params): PredictionCardUsage;
```

默认发放：

```text
每个账号：
- 免亏卡 x1
- 双倍卡 x1
- 反悔卡 x1
- 保底卡 x1
```

### 7.6 前端交互要求

下注确认弹窗中增加竞猜卡选择区域：

```text
不使用竞猜卡
免亏卡 x1
双倍卡 x1
反悔卡 x1
保底卡 x1
```

规则：

1. 默认选择“不使用竞猜卡”。
2. 如果用户没有可用卡，则灰色不可选。
3. 如果该比赛已使用过一张卡，则不可再选。
4. 选择卡片后需要显示规则说明。
5. 提交前二次确认：
   - “本场将使用「免亏卡」，使用后不可撤回，确认继续吗？”

### 7.7 排行榜公平展示

排行榜统计中建议增加：

```text
使用竞猜卡次数
竞猜卡带来的收益修正
未使用竞猜卡收益
实际最终收益
```

第一版不一定在主榜全部展示，但后台和用户详情页必须能查询。

## 八、功能模块四：比赛详情页优化

### 8.1 功能目标

每场比赛做成独立详情页，让用户不仅能下注，还能查看赛前信息、群友分布、AI 分析和赛后复盘。

### 8.2 页面内容

比赛详情页包含：

1. 基础信息
   - 比赛时间
   - 比赛地点
   - 小组/轮次
   - 比赛状态
   - 开赛倒计时
2. 双方球队
   - 队徽
   - 球队名称
   - 世界排名
   - 总身价
   - 近期状态
3. 预计首发
   - 可先使用静态数据
   - 没有数据时显示“暂未公布”
4. 关键球员
   - 每队 3-5 名核心球员
   - 显示位置、俱乐部、身价、标签
5. 历史交锋
   - 双方历史交锋摘要
   - 没有数据可显示“暂无历史交锋数据”
6. AI 赛前分析
   - 一句话看点
   - 三点分析
   - 娱乐倾向
   - 风险提示
7. 群友竞猜分布
   - 支持主队比例
   - 支持平局比例
   - 支持客队比例
   - 大额积分倾向
8. 指数变化
   - 当前指数
   - 变化趋势
   - 可先做静态/简化展示
9. 我的竞猜记录
   - 已下注内容
   - 是否使用竞猜卡
   - 当前状态
   - 预估收益
10. 赛后结果和复盘

- 最终比分
- 我的收益
- 本场最大赢家
- 本场最大冤种
- AI 赛后复盘

### 8.3 状态分段

比赛详情页根据状态展示不同重点：

```text
赛前：信息、AI 分析、竞猜入口
赛中：比分、竞猜锁定、群友分布
赛后：结算结果、赢家榜、AI 复盘
```

## 十七、重点风险点

### 17.1 竞猜卡结算风险

竞猜卡会影响积分，必须保证：

1. 结算逻辑准确。
2. 积分流水准确。
3. 排行榜结果准确。
4. 使用记录可追溯。
5. 结算前自动备份。

### 17.2 称号重复触发风险

同一个称号不能无限重复发放。

要求：

1. 用户已获得的称号不重复创建。
2. 同一比赛触发的称号不重复写入动态。
3. 管理员重新计算时不能制造重复数据。

### 17.3 群内动态过多风险

动态数量会逐渐增加，需要控制展示。

要求：

1. 首页只展示最近 10 条。
2. 后台或详情页可以分页查看。
3. `db.json` 模式下避免无限膨胀，后续可做清理或归档。

### 17.4 AI 调用失败风险

AI API 不稳定时不能影响核心竞猜。

要求：

1. AI 失败不影响下注。
2. AI 失败不影响结算。
3. 前端显示“AI 内容暂不可用”。
4. 后台可重新生成 AI 内容。

### 17.5 `db.json` 写入风险

当前阶段继续使用 `db.json`，必须加强备份。

要求：

1. 启动时备份。
2. 结算前备份。
3. 管理员导出备份。
4. 重要操作写日志。
5. 后续正式部署前再考虑 MySQL 迁移。

## 二十、给 Agent/Codex 的执行要求

1. 每完成一个阶段，必须保证项目能正常启动。
2. 每个阶段完成后执行基础测试。
3. 不要一次性重构所有文件。
4. 不要删除现有核心功能。
5. 不要提交 `.env`、`db.json`、API Key、管理员密码。
6. 所有新增数据结构必须兼容当前 `db.json`。
7. 所有积分变化必须写入流水。
8. 所有管理员关键操作必须写入日志。
9. 竞猜卡系统必须优先保证公平性。
10. AI 功能失败时不能影响竞猜、结算和排行榜。
